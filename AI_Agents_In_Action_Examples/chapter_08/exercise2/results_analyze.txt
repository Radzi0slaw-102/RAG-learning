VII. Summary and remarks
-------------------------------------------------------------------
1. Search latency shows a stark trade-off: TF-IDF (avg 1.78 ms) and Word2Vec
(avg 0.44 ms) are essentially free computationally, while BERT API averages
179.96 ms - roughly 100-400x slower. This gap comes from BERT API relying on
a network call to an external inference endpoint, whereas TF-IDF and Word2Vec
run entirely locally against precomputed vectors. This makes the comparison
somewhat unfair to BERT API: it is penalized by network overhead unrelated to
the quality of its embeddings, and it also risks external failures (timeouts,
rate limits) that the local methods cannot experience.

2. Regarding quality, Word2Vec's high average score (0.80) is misleading. It
never truly fails to produce a "confident-looking" number for multi-word
queries (word-vector averaging smooths over gaps), but drops to 0.00 on five
single-word queries whose words weren't in the training vocabulary (e.g.
"monarch", "laziness", "football", "void").
TF-IDF shows the same behavior (4 zero-scores)since it depends on exact vocabulary overlap.
BERT API never returns 0.0000 - it always produces a semantically meaningful
(if sometimes weak) score, since it generalizes beyond the local vocabulary.

3. Query length matters: for short phrases, all three methods usually agree
(e.g. "royal morning", "azure ceiling"). For full sentences, agreement drops
as paraphrased or reordered content confuses TF-IDF's bag-of-words approach.
For single words, TF-IDF/Word2Vec either overfit to one keyword's context or
fail outright, while BERT API remains the only method producing sensible
results for abstract or unseen terms like "void" or "football".

Overall:
- TF-IDF is fast and interpretable but fragile for paraphrases;
- Word2Vec is fastest and often accurate but binary in failure mode;
- BERT API is the most robust semantically but at a heavy, network-bound latency
cost, and comparing it head-to-head on raw speed against local methods isn't
entirely fair.
I should apologize, as I have set the target to have at least one method be
called from external API, but I didn't consider network speed influence and
I assumed both TF-IDF and Word2Vec are too simple to meet my criteria.
